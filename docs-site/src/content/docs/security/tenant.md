---
title: "多租户设计"
---

微服务拆分后，租户实现的核心是 tenantId 全链路可信传递和各服务本地强制隔离。

租户体系和权限体系需要分开理解：租户体系回答“用户和数据属于哪个租户、租户是否可用、数据如何隔离”；权限体系回答“用户在当前租户下能访问哪些菜单、接口和字段”。

## 核心原则

- 租户生命周期由租户服务管理。
- 用户与租户关系由 IAM 管理。
- Gateway 注入可信租户上下文。
- 每个业务服务按 tenantId 隔离自己的数据。

## 租户和权限的边界

| 问题 | 归属 | 示例 |
| --- | --- | --- |
| 租户是否存在、是否启用 | `xuan-tenant` | 租户被停用后不能登录和访问业务接口 |
| 用户能否进入某个租户 | `xuan-iam` | 用户 A 是否属于租户 T |
| 用户在租户内有哪些角色 | `xuan-iam` | 用户 A 在租户 T 是销售主管 |
| 角色有哪些权限 | `xuan-iam` | 销售主管拥有 `sales:audit` |
| 业务数据属于哪个租户 | 各业务服务 | 销售单、商品、库存都有自己的 `tenant_id` |

租户体系不替代权限体系。一个用户属于某个租户，只代表可以进入该租户，不代表能访问所有菜单和接口。

## tenantId 传递链路

推荐链路：

```text
登录选择租户 -> IAM 签发 token -> Gateway 校验 token 与租户状态 -> 注入请求头 -> 业务服务读取 TenantContext -> MyBatis/JPA 自动追加 tenant_id 条件
```

内部请求头建议：

| Header | 说明 |
| --- | --- |
| `X-Tenant-Id` | 当前租户 ID，由 Gateway 注入 |
| `X-User-Id` | 当前用户 ID |
| `X-User-Type` | 用户类型，例如 platform_admin、tenant_user |
| `X-Trace-Id` | 链路追踪 ID |
| `X-Source-Service` | 内部调用来源服务 |

外部客户端不能直接信任自己传入的 `X-Tenant-Id`。Gateway 必须根据 token 和用户租户关系重新计算可信租户上下文。

## 平台管理员与租户管理员

| 角色 | 说明 | 数据范围 |
| --- | --- | --- |
| 平台管理员 | 管理平台级租户、套餐、系统配置 | 可访问全局管理数据，访问租户业务数据必须有明确审计 |
| 租户管理员 | 管理当前租户内用户、角色、基础资料 | 只能访问当前租户 |
| 租户用户 | 普通业务用户 | 只能访问当前租户授权范围内数据 |

平台管理员不能默认绕过所有业务隔离。需要查看租户业务数据时，应通过“代入租户上下文”或“平台审计授权”进入，并记录审计日志。

## 表设计规则

业务服务中的表默认需要 `tenant_id`。

| 表类型 | 是否需要 tenant_id | 示例 |
| --- | --- | --- |
| 业务主表 | 是 | 商品、客户、销售单、采购单 |
| 业务明细表 | 是 | 销售单明细、库存流水 |
| 租户配置表 | 是 | 租户打印配置、租户业务参数 |
| 全局字典表 | 可不需要 | 系统内置枚举、全局国家地区 |
| 平台租户表 | 可不需要 | 租户主表本身 |
| 审计日志 | 建议需要 | 操作日志、登录日志，平台操作可为空或使用平台租户标记 |

租户内唯一索引必须包含 `tenant_id`：

```sql
unique key uk_product_code_tenant (tenant_id, product_code)
```

禁止只用 `product_code` 做全局唯一，除非该数据确实是平台全局数据。

## 服务内强制隔离

每个业务服务必须具备：

- `TenantContext`：从可信请求头或内部调用上下文读取当前租户。
- 数据访问拦截器：自动给查询追加 `tenant_id` 条件。
- 写入填充：创建业务数据时自动写入 `tenant_id`。
- 越权校验：更新、删除、审核时必须校验目标数据属于当前租户。
- 测试用例：验证不同租户不能查询、更新、删除彼此数据。

## 服务间调用

服务间调用必须继续传递租户上下文：

```text
xuan-sales -> xuan-inventory
  X-Tenant-Id: 当前租户
  X-Source-Service: xuan-sales
  X-Trace-Id: 当前链路
```

被调用服务不能因为请求来自内部服务就跳过租户校验。内部服务只解决“谁调用”，不解决“可以访问哪个租户的数据”。

## Query Service 租户隔离

`xuan-query` 会聚合多个服务的数据，是最容易串租户的位置。

要求：

- 读模型表必须包含 `tenant_id`。
- 消费业务事件时使用事件中的 `tenantId` 写入读模型。
- 查询报表时必须按当前 `tenantId` 过滤。
- 跨租户平台报表必须使用独立接口、独立权限和审计记录。

## 租户停用拦截

租户停用后：

| 场景 | 处理 |
| --- | --- |
| 登录 | IAM 拒绝签发当前租户 token |
| 已登录 token | Gateway 校验租户状态，必要时拒绝请求 |
| 业务接口 | 业务服务本地可缓存租户状态并兜底校验 |
| 异步消费 | 消费端根据事件语义决定停止处理或只记录归档 |
| 定时任务 | 跳过停用租户，除非是平台维护任务 |

租户停用不应物理删除业务数据。

## 落地难度判断

微服务拆开后，租户实现会更严格，但不一定更困难。真正的难点不是代码量，而是边界一致：

- 每个服务都要有统一的租户上下文组件。
- 每个服务都要有数据访问层租户拦截。
- 每个服务都要有跨租户回归测试。
- 事件、读模型、导出、审计都要携带 `tenantId`。

如果这些规则从一开始就写进公共模块和测试模板，拆成 14 个服务仍然可控；如果每个服务自己随便实现，后期会非常难排查串租户问题。
