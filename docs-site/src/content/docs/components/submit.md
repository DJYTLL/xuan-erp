---
title: "提交功能"
---

本文记录业务提交功能的统一约定，作为类似 RuoYi 功能组件文档的起点。

## 1. 使用场景

用于保存、提交、审核、确认等需要防重复、校验权限和记录审计的业务操作。

## 2. 前端调用方式

- 写操作必须通过统一 API 封装调用。
- 保存、提交、审核成功后使用统一后续操作弹窗。
- 审核流程只确认一次：确认后先静默保存，再调用审核接口。
- 风险操作必须二次确认并填写原因。

## 3. 后端接口约定

- Controller 必须加 `@PreAuthorize`。
- Controller 入参转换为 Command，再交给 ApplicationService。
- 写操作支持幂等键，避免重复提交。
- ApplicationService 负责事务边界、幂等、审计和领域对象调用。
- 新增、编辑、删除、审核、红冲等关键操作必须记录审计日志。
- 删除接口统一要求删除原因。
- 不暴露恢复接口，除非业务明确设计恢复流程。

## 4. 权限要求

- 前端入口显示、接口请求前置判断、后端 Controller/ApplicationService 校验都应使用当前业务自己的权限。
- 跨业务单据引用数据时，权限归属当前业务场景，不复用被引用模块的查看权限。
- super admin 只能旁路鉴权，不能省略权限定义。

## 5. 租户隔离要求

- Service/ApplicationService 层必须校验租户上下文。
- 查询默认带 `tenant_id` 和 `deleted_at` 条件。
- 没有数据库外键时，完整性校验必须放到应用层或领域层。

## 6. 异常处理

- 返回格式统一为 `{ code, message, data }`。
- 错误响应必须包含业务错误码，不能只返回异常文本。
- 页面错误提示统一使用 `useApiError()`。

## 7. 注意事项

- 不相关写操作不能为了“少请求”强行塞进一个接口。
- 当前业务动作必须展示的数据，由后端一次性聚合返回，禁止前端或后端 N+1 次请求。
- 审核、复制、红冲等依赖详情状态的按钮要防闪现：初始化阶段用禁用占位按钮。
