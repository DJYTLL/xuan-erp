---
title: "接口契约规范"
---

接口契约用于统一前后端调用和服务间 HTTP 调用的行为。微服务拆分后，接口文档必须明确路径、权限、租户、幂等、错误码和审计字段。

接口层属于 DDD 的 `interfaces` 层，只负责协议适配和 DTO 转换，不承载核心业务规则。

## 基础规则

- 所有后端接口统一通过 Gateway 暴露给前端。
- 内部服务间调用优先使用 OpenFeign，并引入 `spring-cloud-starter-loadbalancer` 负责客户端负载均衡。
- 服务间调用必须通过服务间鉴权传递调用身份。
- Controller 只做协议适配，不写业务逻辑。
- Controller 必须加 `@PreAuthorize`。
- Controller 入参应转换为应用层 Command 或 Query，再交给 ApplicationService 处理。
- 所有写接口必须记录操作者、租户、TraceId 和业务审计信息。
- 分页、排序、筛选参数保持统一命名。
- 错误响应必须包含业务错误码，不能只返回异常文本。
- Response DTO 不直接暴露数据库对象或领域对象内部结构。
- MyBatis Mapper、PO/DO、Domain Entity 不直接作为接口响应。
- 新增、编辑、删除、审核、红冲等关键操作必须记录审计日志。
- 删除接口统一要求删除原因。
- 业务删除默认逻辑删除，使用 `deleted_at`。
- 不暴露恢复接口，除非业务明确设计恢复流程。
- 写操作支持幂等键，避免重复提交。

## 一次请求完成原则

当前页面首屏或当前业务动作必须展示的数据，应由后端一次性聚合返回；禁止因为明细行、关联 ID 或字典项数量增长，造成前端或后端 N+1 次请求。

典型场景：销售单详情中已经查到商品 `productId` 后，后端应批量查询商品快照、仓库、单位、规格等展示字段，并在销售单响应 DTO 中一次性返回给前端。前端不应拿到每一行商品 ID 后，再逐行请求商品名称、商品仓库或库存信息。

接口设计要求：

- 列表页和详情页的 Response DTO 要包含当前页面直接展示所需的名称、编号、状态、仓库、单位等快照字段。
- 后端需要跨服务补齐数据时，应优先使用批量接口，例如按 `productIds` 一次查询商品快照，按 `warehouseIds` 一次查询仓库快照。
- 禁止在循环中逐条调用远程服务、逐条查数据库或让前端逐条补查展示字段。
- 对于可能缺失的关联数据，后端要给出稳定降级结果，例如返回历史快照、占位名称或明确的缺失状态，不能把补齐责任甩给前端。
- 写接口仍以业务命令为主，不为了减少请求把多个无关写操作强行塞进一个接口。

边界和例外：

- 很重的数据，例如历史轨迹、审计日志、附件列表，可以点开时再懒加载。
- 实时变化的数据，例如库存可用量，如果特别敏感，可以单独刷新，但仍要支持批量查询。
- 超大明细不能无限一次性返回，必须有分页、分段加载或单次数量上限。
- 不相关的写操作不能为了“少请求”强行塞进一个接口，应保持业务命令边界清晰。

销售单详情响应示例：

```json
{
  "id": 10001,
  "orderNo": "SO202606100001",
  "items": [
    {
      "productId": 20001,
      "productCode": "P-001",
      "productName": "标准商品",
      "warehouseId": 30001,
      "warehouseName": "成品仓",
      "unitName": "件",
      "quantity": 10
    }
  ]
}
```

## 路径规范

| 类型 | 示例 |
| --- | --- |
| 前端访问 | `/api/products` |
| 内部服务 | `/internal/products/{id}` |
| 管理接口 | `/admin/permissions/sync` |
| 健康检查 | `/actuator/health` |

## 响应结构

```json
{
  "code": "SUCCESS",
  "message": "OK",
  "data": {},
  "traceId": "01J00000000000000000000000"
}
```

## 幂等要求

销售审核、采购审核、库存扣减、收付款、红冲、打印提交等写操作必须考虑幂等。推荐使用业务单号、请求幂等键、状态机和唯一约束共同保证。

## 删除要求

删除、作废、红冲等风险操作必须带原因，并记录操作者、租户、TraceId、业务单号和操作前后状态。

默认不提供恢复接口。确实需要恢复时，必须先明确恢复流程、权限、审计、数据一致性和唯一索引占用问题。
